# /clipping-research — 六维深入理解框架

对指定的 clippings 进行深度阅读和调研，输出结构化理解到对话上下文。不写 vault，不做 decompose。

**定位**：三步流水线的第二步。
1. `obsidian-deep-research "处理 clippings"` → 聚簇扫描，用户选择一簇
2. **`/clipping-research <文件列表>`** → 六维深入理解（本 command）
3. `obsidian-deep-research "decompose 上面的内容"` → 从上下文 decompose

## 输入

`$ARGUMENTS` 为以下之一：
- 文件路径列表（空格分隔）：`Clippings/xxx.md Clippings/yyy.md`
- 主题关键词：`harness engineering`（自动搜索未处理的相关 clippings）
- 用户直接粘贴的文章内容（含或不含 URL）

如果是关键词，先用 Grep 在 `Clippings/` 和 `10-Inbox/Clippings/` 中搜索匹配文件，展示候选列表让用户确认。

## 写作原则

本 skill 的输出面向**将来重新阅读的自己**——一个对主题可能已经遗忘了细节的读者。每个维度都要让读者在没有原文辅助的情况下能独立理解。

核心写作要求：
- **先解释再分析**：每个维度开头用 2-3 句通俗语言概述这个维度要回答什么问题、得出了什么结论，然后再展开细节
- **解释专业术语**：首次出现的专业术语必须用括号或短句解释其含义，不能假设读者记得
- **说明关联**：提到前置概念、相邻领域时，说明它们和当前主题的具体关系（"X 是 Y 的底层机制，因为..."），而不是只列名词
- **精选而非穷举**：案例、来源、数据点都选最有说服力的少数几个，深度展开，而非浅尝辄止地列很多

## 执行流程

### Step 1: 深入理解（Comprehend）

逐篇读取所有 clippings，提取：

| 文件 | 主题 | 关键声明 | 引用的外部来源 |
|------|------|----------|----------------|

然后追溯 top 3-5 个外部来源：
- 用 `mcp__exa__web_search_exa` 搜索原始来源
- 用 `defuddle` 或 `WebFetch` 读取原文
- 对比 clipping 与原文的差异：是否过度简化？遗漏了哪些关键 nuance？

**输出要求**：
1. 先用 3-5 句话回答"这篇文章到底在说什么"——用读者能直接理解的语言，不依赖原文术语
2. 然后给出核心概念图谱（用缩进或 ASCII 树结构展示概念间的层级和因果关系）
3. 最后列出 clipping 与原文对比中发现的差异（过度简化 / 遗漏 / 错误）

### Step 2: 具体案例理解（Ground）

用行业实际案例辅助理解抽象概念。案例的价值在于把抽象概念锚定到具体实践，帮助判断 clipping 中的观点是否有广泛适用性。

**数量限制：最多 3 个案例**，选择标准：
1. 优先使用 clipping 中**明确提到**的案例——搜索一手资料
2. 如果 clipping 案例不足，补充搜索**最知名/最有代表性**的同领域实践

**每个案例的描述深度**（不是一两句话概括，而是让读者真正理解发生了什么）：
- **背景**（2-3 句）：谁在什么场景下遇到了什么问题
- **做法**（3-5 句）：具体采取了什么方案，技术细节到什么程度、关键设计决策是什么
- **结果**（1-2 句）：量化效果（如有）或定性评价
- **可复用模式**（1 句）：从这个案例中抽象出的通用原则

### Step 3: 最佳实践查找（Validate）

查找行业/社区是否已形成共识：

- `mcp__exa__web_search_exa` 搜索 "best practices" + 主题关键词
- 可选：`/last30days` 查社区信号（X、HN、Reddit 近期讨论）
- 查找相关 GitHub 项目/框架

输出三类信号：
- **共识点** — 多个独立来源验证的观点（注明至少 2 个来源）
- **争议点** — 来源之间有分歧（说明双方各自的论据）
- **空白点** — 无人讨论（可能是 clipping 独创或过于小众）

### Step 4: 领域定位（Contextualize）

在更大的知识领域中定位这个话题：

- **前置概念**：这个话题建立在哪些已有概念之上？对每个概念用 1 句话说明它和当前主题的关系（"A 是 B 的底层机制，因为..."）
- **相邻领域**：和哪些看似不同但有关联的领域有交叉？说明交叉点在哪里
- **演变趋势**：之前的范式是什么？当前处于什么阶段？可能演变成什么？用时间线形式展示
- **vault 关联**：用 Grep 检查 vault 中是否已有相关笔记，标注**重叠**（已有的知识）和**增量**（本次新增的知识）

### Step 5: 反面审视（Challenge）

主动寻找反对观点和局限性，避免确认偏差：

- 谁在反对这个观点？理由是什么？
- 在什么条件下这个观点不成立？
- 文章中是否有逻辑跳跃或未说明的假设？
- 数据是否 cherry-picked？是否存在幸存者偏差？
- 作者是否有利益冲突？（如向量数据库公司论证长上下文不可靠）

搜索反对观点时，用 `mcp__exa__web_search_exa` 搜索 "criticism" / "limitations" / "problems with" + 主题关键词。

**信噪比要求**：只提炼观点的核心论据（每个反对观点 2-3 句），不要大段引用他人文章或 LLM 对话记录。

### Step 6: 关键数据校验（Quantify）

提取并验证文章中的量化数据：

| 数据点 | 来源 | 验证状态 | 验证方法 |
|--------|------|----------|----------|
| 示例数据 | 原始论文 | ✅ 已确认 | 追溯到论文 Table 3 |
| 示例数据 | 仅在 clipping 中 | ⚠️ 未验证 | 搜索未找到原始出处 |

验证状态：
- ✅ 已追溯到**独立来源**并确认（说明追溯到的具体位置）
- ⚠️ 仅在 clipping 中出现，未找到原始来源
- ❌ 与原始来源数据不一致（说明差异）

注意："原文也这么说"不算独立验证。✅ 要求找到 clipping 引用的**上游来源**（论文、官方文档、原始实验）来确认。

## 输出格式

所有内容输出到对话上下文，使用以下结构：

```
## 理解：{主题}

### 一句话总结
{用一句话概括这个主题的核心 takeaway，让读者立即知道这篇文章最重要的结论是什么}

### 核心概念（Comprehend）
{先 3-5 句通俗解释 → 概念图谱 → 原文对比发现}

### 行业案例（Ground）
{最多 3 个案例，每个案例：背景→做法→结果→模式，描述要足够详细让读者理解发生了什么}

### 最佳实践与共识（Validate）
{共识点（附来源）/ 争议点（附双方论据）/ 空白点}

### 领域定位与演变（Contextualize）
{前置概念（解释关联）/ 相邻领域（说明交叉点）/ 演变时间线 / vault 关联（重叠+增量）}

### 反对观点与局限（Challenge）
{精炼的反对论据 / 适用边界 / 逻辑检验 / 利益冲突审查}

### 关键数据（Quantify）
{数据表 + 验证状态 + 验证方法}

### 知识 gap
{本次调研未能回答的问题，建议后续方向}
```

## 关键约束

- **不写 vault** — 所有内容留在对话上下文，供后续 decompose 使用
- **不提出 decompose 方案** — 那是 obsidian-deep-research 的职责
- **只负责"理解"，不负责"组织"** — 理解层和组织层分离是这个流水线的核心设计
- 每个 step 完成后简要报告进度，全部完成后输出完整结构化理解
